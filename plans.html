By the end of term 1 we will deliver a prototype that has the basic functionality of pulling in tweets from the live-stream and displaying the tweets with the sentiment labelled to the user. The user also has the option to filter out the data via the 3 available categories of tweet sentiment: Good, Bad and Neutral. The image below shows a mock-up of the final delivarable for this project.

  • The next step of extending this project would be to develop an algorithm that given a text, classifies it into one of the three possible categories. For this we will use Natural Language Processing Libraries and Techniques (further details of this is availble in the Research section). We will refine the algorithm to the point where we have an acceptable error rate, possibly 10%-20%. If the error-rate is achieved within the sprint deadline we will tweak it further to increase the accuracy else we will move on.

  • Currently the prototype only works with tweets using the twitter API, this functionality will be extended to include posts/statuses form other social media platforms. Further data needs to be attached to each entry in the output to include the name of the platform from which it was fetched etc. When we are able to display information form a range of different platforms we will either need to devise an algorithm to group the data in a useful way i.e. clustering the data from different social media and sort them in a pre-defined order; or we will agree on a protocol to display the data from a certain perspective.

  • There will be a resource re-direction mechanism built into the back-end of the project to only display the relevant information to the person using the system.This means that a further classification system is to be configured which clusters data into a given number of fields (ex: Train operator, Janitor, Toilet manager etc) and dis-patches the tweets to them. The most sensible way to do it, is through a login screen where the credentials would be matched against a secure database and upon validation, the user will be re-directed to the page specific to the user. This brings up security concerns such as securing the database from intrusion as senstive data is available upon login. Further research will be carried out to establish what security measures we will implement and provide.

  • The protype that we have build is running using PHP and was built like that because we did not want to overcomplicate some features of the implementation. We aim to do the final application using Django, but if it happens that a PHP solution gives us a smoother and less complicated alternative, we will use that instead.
